10. 模型输出里的轻量前端模板不再直接裸露：`<p style>`、`<content>`、`<br>` 等白名单标签会被安全渲染或剥离；脚本、iframe、任意 CSS / JS 仍不执行。

## 1. 本次调研结论

一句话结论：

Narra 现在不是「功能缺很多」的问题，而是「已有能力很强，但产品入口、信息架构、资源生态和用户可感知完成度还没有叙说那么完整」。

更具体地说：

1. Narra 的底层能力已经很深。
   - 多 Provider。
   - 普通聊天。
   - 沉浸扮演。
   - 角色 / 助手。
   - 世界书。
   - 长期记忆。
   - 上下文日志。
   - 用户面具。
   - 查手机。
   - 朋友圈。
   - 日记。
   - 视频通话。
   - 线上 / 线下 / 长文模式。
   - 无背景皮肤。
   - 屏幕翻译。
   - 上下文导入导出。

2. 叙说更强的是「产品化完整度」。
   - 首页和功能入口更集中。
   - 社区资源更外显。
   - 角色、记忆、世界书、模板、音乐、小说、信箱、论坛等模块被包装成一个完整生态。
   - 用户不需要理解太多底层术语，也能知道下一步点哪里。

3. Narra 不应该直接复制叙说。
   - 叙说偏「微信式 / Web App / 社区生态 / 多功能平台」。
   - Narra 应该偏「原生 Android / 沉浸式角色关系 / 私人手机感 / 深上下文工程 / 可配置创作工具」。

4. Narra 最适合吸收的是：
   - 功能入口的完整组织。
   - 资源生态的分层。
   - 角色关系型玩法。
   - 信箱、主题包、资源包、社区分享这些增强沉浸感的模块。

5. Narra 当前最不建议做的是：
   - 直接把主界面改成叙说同款。
   - 为了像叙说而新增大量半成品模块。
   - 默认开放 HTML / JS 模板执行。
   - 把「场景」继续作为用户第一心智。
   - 过早建设大社区。

## 3. 叙说·春信产品画像

基于此前对 `https://cx.xushuo.cc/` 的调研，叙说整体更像一个面向角色互动用户的 Web 端完整平台。

它不是单一聊天工具，而是把以下能力包装成一个连续体验：

- AI 连接配置。
- 角色聊天。
- 角色卡 / 预设 / 世界书。
- 长期记忆。
- 社交式界面。
- 朋友圈 / 广场 / 论坛。
- 信箱。
- 音乐。
- 小说。
- 资源分享。
- HTML 模板。
- 主题 / 外观。
- PWA / 离线缓存。

它的优势不一定是每个模块都比 Narra 更深，而是用户会感觉「这个软件什么都有，而且入口完整」。

这就是 Narra 最值得学的地方：

不要只把能力藏在高级设置、Prompt、Repository 或单独页面里，而要让用户在主流程里感知到这些能力。

## 4. 叙说值得吸收的核心能力

### 4.1 完整的信息架构

叙说把功能组织成一种「像日常社交软件一样自然」的结构：

- 聊天。
- 角色。
- 社区。
- 设置。
- 资源。
- 扩展玩法。

这会降低用户学习成本。

Narra 已经有 `ImmersivePhoneShell`，里面有：

- 消息。
- 通讯录。
- 广场。
- 我。

这说明 Narra 已经开始往类似信息架构走了。后续不需要从零做，而是应该打磨它。

### 4.2 角色生态感

叙说的角色不是单纯的「助手配置」，而是被包装成可被浏览、收藏、分享、沉浸互动的对象。

Narra 当前的 `Assistant` 和 `RoleplayScenario` 已经支持：

- 名称。
- 描述。
- 系统提示词。
- 场景 / 聊天背景。
- 开场白。
- 示例对话。
- 作者备注。
- 关联世界书。
- 关联记忆。
- 角色头像。
- 用户形象覆写。
- 角色形象覆写。

底层字段已经足够丰富。缺的主要是产品表达：

- 角色名片。
- 角色主页。
- 标签筛选。
- 最近互动。
- 可导入导出。
- 可分享。
- 可组合世界书 / 记忆 / 主题。

### 4.3 社交拟态

叙说的社交拟态很明显，包括聊天、朋友圈、广场、信箱等。

Narra 也已经有：

- `PHONE_CHECK`
- `MOMENTS`
- `ROLEPLAY_DIARY`
- `ROLEPLAY_VIDEO_CALL`
- 线上手机聊天模式。
- 语音、转账、邀约、礼物、委托、惩罚等快捷玩法。

这其实已经很有特色。

Narra 不需要照搬叙说的微信式壳，而应该把这些能力做成「角色的私人手机」。

关键区别：

- 叙说像一个平台。
- Narra 可以像一部属于角色与用户关系的手机。

### 4.4 资源和社区

叙说的强项是社区资源外显。

Narra 当前更像本地工具，资源能力主要体现在：

- 助手。
- 世界书。
- 记忆。
- 上下文导入导出。
- 用户面具。
- 设置。

这类能力已经具备做资源包的基础。后续可以先做本地资源库和导入导出，再考虑线上社区。

推荐路径：

1. 本地资源包。
2. 文件导入导出。
3. 用户手动分享。
4. 可选资源市场。
5. 真正社区。

不要一开始就做大社区。那会把产品、审核、账号、后端、运营复杂度一起引进来。

### 4.5 信箱

叙说的信箱模块对 Narra 很值得学。

原因是信箱天然适合 Narra 的沉浸扮演：

- 可以写给角色。
- 可以收到角色来信。
- 可以作为剧情节点。
- 可以承载长期关系变化。
- 可以和日记、记忆、世界书联动。

它比音乐、小说、论坛更适合作为 Narra 的中期新增模块。

推荐把「信箱」设计成 Narra 特色：

- 不是普通站内信。
- 不是社区私信。
- 而是「角色关系里的信件系统」。

可以包含：

- 收件箱。
- 草稿箱。
- 未寄出的信。
- 定时信。
- 角色主动来信。
- 剧情触发信。
- 信件转长期记忆。
- 信件引用世界书。

### 4.6 主题和模板

叙说有更外显的模板与主题能力。

Narra 当前已经有 `RoleplayNoBackgroundSkinPreset`：

- 微信。
- QQ。
- Telegram。
- KakaoTalk。
- Retro。
- 波点复古。
- 复古像素。
- 美丽少女。
- 哥特夜诗。
- Y2K。
- iMessage。
- 液态玻璃。

这说明 Narra 已经拥有「皮肤系统雏形」。

但当前默认值是 `WECHAT`。如果想保留 Narra 自己特色，建议后续新增一个默认的 `NARRA` 或 `NARRA_GLASS` 皮肤，把微信变成可选项。

也就是说：

- 微信皮肤可以保留。
- 但 Narra 默认不应该等同微信。
- Narra 应该有自己的默认视觉。

## 5. Narra 当前真实现状

以下判断基于源码，不基于参考图片。

### 5.1 技术栈

Narra 是一个原生 Android 项目。

主要技术：

- Kotlin。
- Jetpack Compose。
- Material 3。
- Navigation Compose。
- ViewModel。
- StateFlow。
- DataStore。
- Room。
- Retrofit。
- OkHttp。
- Gson。
- KSP。
- Baseline Profile。

这和叙说 Web App 的技术路径完全不同。

因此 Narra 的优势不该只和网页 UI 比，而应该发挥原生 App 能力：

- 系统分享。
- 本地文件。
- Android 通知。
- 前台服务。
- 无障碍屏幕翻译。
- 原生存储。
- APK 发布和应用内更新。
- 更稳定的长会话体验。

### 5.2 当前产品骨架

当前主要路由包括：

- `HOME`
- 普通聊天。
- 设置。
- Provider 设置。
- 模型管理。
- 搜索工具。
- 助手管理。
- 世界书。
- 记忆管理。
- 上下文导入导出。
- 上下文日志。
- 屏幕翻译。
- 沉浸扮演。
- 聊天管理。
- 聊天编辑。
- 沉浸聊天。
- 沉浸设置。
- 阅读模式。
- 日记。
- 视频通话。
- 查手机。
- 朋友圈。

这说明 Narra 的模块深度已经超过一个普通聊天 App。

### 5.3 首页

`HomeScreen.kt` 当前首页偏工具型：

- 欢迎回来。
- 当前配置状态。
- Base URL。
- API Key。
- 模型。
- 进入聊天。
- 进入设置。
- 沉浸扮演。

这个首页对开发者和高级用户清晰，但对角色互动用户略显「配置优先」。

如果对标叙说，建议：

- 保留配置检查。
- 但把主入口更偏向「开始互动」。
- 配置状态可以降级为小提示。
- 沉浸扮演入口可以更强。

建议方向：

- 首页不要做成营销页。
- 也不要变成叙说首页克隆。
- 可以做成 Narra 的「控制台 / 私人空间」。

### 5.4 沉浸主壳

`ImmersivePhoneShell.kt` 是 Narra 非常关键的现有资产。

它已经有四个 Tab：

- 消息。
- 通讯录。
- 广场。
- 我。

消息页已有：

- 会话列表。
- 最近消息。
- 置顶。
- 免打扰。
- 清空。
- 删除。
- 新建聊天。
- 聊天管理。

通讯录页已有：

- 角色列表。
- 联系人卡片。
- 从角色创建聊天。

广场页已有：

- 朋友圈。
- 查手机。
- 日记本。
- 视频通话。

我的页已有：

- 用户资料 / 面具入口。
- 设置入口。
- 聊天和角色数量。

这个结构非常接近叙说的产品感，但目前更像「功能已经摆上去」，还需要进一步打磨：

- 空状态更有沉浸感。
- 会话卡片更像角色关系，而不是普通列表。
- 通讯录可以增加角色分组 / 标签。
- 广场可以改名或拆分，避免像公共社区但实际是本地功能。
- 我的页可以承载用户人格、默认面具、资源包、备份。

### 5.5 聊天管理

`RoleplayScenarioListScreen.kt` 当前文案已经开始从「场景」转为「聊天资料」。

这和此前改造计划一致：用户心智应该是「聊天 / 角色关系」，不是「场景」。

当前卡片支持：

- 背景图。
- 渐变兜底。
- 标题。
- 绑定助手。
- 可继续状态。
- 编辑。
- 播放 / 进入。
- 交互模式标签。

建议继续强化：

- 把内部 `Scenario` 概念继续藏起来。
- 对用户统一叫「聊天」或「聊天资料」。
- 卡片上显示关系状态、最近动态、模式、记忆状态。

### 5.6 沉浸聊天页

`RoleplayScreenContent.kt` 当前已经有很强的沉浸能力：

- 背景图。
- Haze 玻璃效果。
- 角色和用户人像。
- 说话者高亮。
- 场景情绪颜色。
- 标题栏可收起。
- 身份条。
- 状态条。
- 长期记忆提议卡片。
- AI 帮写。
- 快捷功能。
- 阅读模式。
- 上下文治理。

这部分是 Narra 应该保留并放大的核心特色。

叙说可以提供功能参考，但 Narra 的沉浸聊天页不应该被改成普通 Web 聊天窗口。

### 5.7 线上手机模式

`RoleplayOnlinePhoneContent.kt` 是 Narra 的另一块关键资产。

它已经支持：

- 手机聊天气泡。
- 快捷功能。
- 日记。
- 语音。
- 查手机。
- 转账。
- 邀约。
- 礼物。
- 委托。
- 惩罚。
- 朋友圈。
- 视频通话。
- 引用消息。
- 撤回消息。
- 截图聊天。

这已经不是普通聊天功能，而是「互动玩法层」。

后续如果借鉴叙说，应该把这些能力包装得更像一套完整关系系统：

- 每个玩法有入口、有记录、有影响。
- 玩法结果进入记忆 / 日记 / 手机快照。
- 用户能回看角色关系变化。

### 5.8 皮肤系统

`RoleplayNoBackgroundSkinSettings.kt` 和 `RoleplayNoBackgroundSkin.kt` 表明 Narra 已经有多套聊天皮肤。

当前能力：

- 皮肤预设。
- 气泡宽度。
- 圆角。
- 横向 padding。
- 纵向 padding。
- 气泡尾巴。
- 每套皮肤独立背景 / 面板 / 描边 / 重点色 / 气泡色 / 文本色。

这是做「主题包」的基础。

建议补齐：

- Narra 默认主题。
- 主题预览。
- 主题导入导出。
- 按聊天绑定主题。
- 按角色绑定主题。
- 主题包和角色卡一起分享。

### 5.9 设置系统

Narra 当前设置能力丰富，但有些入口仍偏工程化。

已有：

- API Provider 管理。
- 连接设置。
- 模型管理。
- 搜索工具。
- 助手配置。
- 世界书。
- 记忆管理。
- 上下文导入导出。
- 屏幕翻译。
- 应用更新。
- 显示与行为。
- 上下文日志。

现有计划 `tavo_alignment_plan.md` 已指出：

- 需要把 API 连接合并到提供商页。
- 删除或弱化单独的连接设置。

这个方向是对的。

对标叙说时，设置页应该从「参数堆叠」变成「能力分组」：

- 模型与连接。
- 角色与资料。
- 沉浸与外观。
- 记忆与上下文。
- 导入导出。
- 工具能力。
- 版本与更新。

### 5.10 记忆和上下文

Narra 的记忆和上下文能力非常强。

从源码和文档看，已经包括：

- 全局记忆。
- 助手记忆。
- 会话记忆。
- 角色长期记忆。
- 情景记忆。
- 剧情总结。
- 记忆提议。
- 摘要刷新。
- 上下文日志。
- 世界书扫描。
- Prompt 组装日志。

这是 Narra 明显强于许多同类工具的地方。

但问题是：

用户未必感知得到。

建议把记忆能力产品化：

- 每个角色有「记忆档案」。
- 每段聊天有「关系时间线」。
- 记忆提议可以从聊天页一键处理。
- 日记、信箱、手机快照都可以回流记忆。

### 5.11 屏幕翻译

屏幕翻译是 Narra 与叙说差异最大的能力之一。

它不属于角色扮演核心，但属于原生 Android 工具优势。

建议保留，并把它定位为 Narra 的「通用 AI 工具底座」：

- 普通聊天负责通用 AI。
- 沉浸扮演负责角色关系。
- 屏幕翻译负责跨 App 辅助。

这会让 Narra 不只是叙说式角色平台，而是更完整的原生 AI App。

## 6. Narra 必须保留的特色

### 6.1 原生 Android 体验

Narra 是原生 Android，不是 Web 套壳。

这意味着它应该保留：

- Material 3 的稳定交互。
- 系统分享。
- 本地文件读写。
- 图片选择和预览。
- 无障碍屏幕翻译。
- 应用内更新。
- 原生导航和状态栏处理。
- 更稳定的长时间使用体验。

### 6.2 深上下文工程

Narra 的核心竞争力之一不是 UI，而是上下文工程。

包括：

- PromptContextAssembler。
- RoleplayPromptDecorator。
- 世界书。
- 记忆选择。
- 摘要。
- 上下文日志。
- 角色长期记忆。
- 手机快照注入。
- 查手机线索注入。

这些能力不要为了做表层功能而削弱。

### 6.3 角色关系系统

Narra 不是普通「机器人聊天」。

从已有代码看，它已经在做角色关系系统：

- 角色资料。
- 用户面具。
- 聊天资料。
- 互动模式。
- 日记。
- 朋友圈。
- 手机快照。
- 视频通话。
- 转账 / 礼物 / 邀约等特殊玩法。

这个方向比单纯仿微信更有价值。

### 6.4 线上 / 线下 / 长文三类体验

当前 `RoleplayInteractionMode` 有：

- `ONLINE_PHONE`：线上模式。
- `OFFLINE_DIALOGUE`：普通对白。
- `OFFLINE_LONGFORM`：长文线下。

再加上 `enableNarration`，可以组合出类似叙说多模式的体验。

建议不要随便新增底层模式。

更好的做法：

- 底层继续保持 3 类。
- UI 层包装成 5 种常用入口。
- 当前 `ImmersivePhoneShell` 已经这样做了。

### 6.5 可读性和沉浸设置

Narra 已有：

- 沉浸模式。
- 边到边。
- 全屏。
- 标准。
- 行距档位。
- 高对比度。
- 身份条。
- 状态条。
- AI 帮写开关。
- 线上心声显示。
- 皮肤细节设置。

这类细节很适合长时间阅读和长剧情。

叙说的完善感可以学，但 Narra 自己这套可读性设置要保留。

## 7. 模块对照分析

### 7.1 聊天主线

叙说：

- 入口明确。

### 7.10 小说

## 8. Narra 当前差距

### 8.1 产品入口差距

Narra 的能力很多，但入口有时分散。

典型表现：

- 首页更像配置页。
- 设置里工程项较多。
- 角色、聊天、助手、场景概念仍需继续统一。
- 一些强功能藏在沉浸聊天里的菜单或设置中。

建议先做入口梳理，不急着新增大模块。

### 8.2 生态感差距

叙说给用户的感觉是：

这里有角色、资源、社区、玩法。

Narra 当前给用户的感觉可能是：

这里有很多高级功能，但我要自己摸索。

解决方式：

- 角色名片。
- 资源库。
- 一键新建聊天。
- 主题包。
- 角色包。
- 世界书挂载提示。
- 记忆档案。

### 8.3 默认视觉识别差距

Narra 已经有多套皮肤，但当前默认无背景皮肤是微信。

这会带来一个问题：

用户可能会以为 Narra 是微信聊天模拟器。

建议：

- 新增 Narra 默认皮肤。
- 微信、QQ、Telegram 等作为可选皮肤。
- 默认视觉体现 Narra 自身：轻玻璃、安静、私密、角色关系感。

### 8.4 社区和分享差距

叙说的资源分享能力更明显。

Narra 当前已有上下文导入导出，但更偏备份迁移。

建议把「导入导出」升级成「资料库」：

- 角色包。
- 世界书包。
- 主题包。
- 记忆包。
- 聊天备份。
- 手机快照包。

### 8.5 新用户引导差距

Narra 有大量高级功能，新用户可能不知道：

- 先建角色还是先建聊天。
- 世界书何时使用。
- 记忆怎么生效。
- 查手机和朋友圈有什么关系。
- 线上 / 线下 / 剧情模式怎么选。

建议增加轻量引导：

- 新建聊天时解释模式。
- 角色页展示下一步。
- 空状态给明确操作。
- 不做大段教程。

## 9. 推荐定位

### 9.1 不推荐定位

不建议把 Narra 定位成：

- 叙说复刻版。
- 微信聊天模拟器。
- 单纯角色卡社区。
- 单纯 Prompt 工具箱。
- 大而全 Web 社区。

### 9.2 推荐定位

建议定位成：

Narra 是一个原生 Android 的沉浸式 AI 角色关系应用，兼具通用 AI 聊天、深上下文记忆、角色世界书和私人手机式互动体验。

更短的产品心智：

「和角色长期相处的一部私人手机。」

这个定位能容纳：

- 消息。
- 通讯录。
- 朋友圈。
- 查手机。
- 日记。
- 视频通话。
- 信箱。
- 记忆。
- 世界书。
- 主题。
- 普通 AI 聊天。
- 屏幕翻译。

也能和叙说区分开。

## 10. 总体改造原则

### 10.1 不照搬 UI

叙说的 UI 可以参考，但 Narra 不应该复制。

尤其是：

- 不强行 Web 化。
- 不强行社区首页化。
- 不把微信风格作为唯一默认。
- 不牺牲原生 Android 交互。

### 10.2 先整理已有能力

Narra 现在已有能力足够多。

优先级应该是：

1. 整理入口。
2. 统一命名。
3. 提升可发现性。
4. 建立资源包。
5. 再新增模块。

### 10.3 概念统一

用户侧建议统一：

- 角色：对应 Assistant 的用户心智。
- 聊天：对应 RoleplayScenario / Session 的用户心智。
- 资料：角色、世界书、记忆、主题、聊天记录等可管理对象。
- 手机：查手机、朋友圈、信箱、视频通话等沉浸玩法集合。

内部代码暂时可以保留 `Scenario`，但 UI 文案继续向「聊天」靠拢。

### 10.4 默认体验要像 Narra

皮肤系统可以支持微信。

但默认体验要像 Narra：

- 私密。
- 沉浸。
- 干净。
- 有角色关系感。
- 有轻玻璃 / Material 3 质感。
- 能长时间阅读。

### 10.5 高风险能力延后

以下能力不要第一阶段做：

- 大社区。
- 论坛。
- 在线资源市场。
- HTML / JS 模板执行。
- 复杂音乐播放器。
- 小说平台。

这些可以做，但需要后端、审核、权限、安全和长期维护。

## 14. 数据与安全建议

### 14.1 不导出敏感配置

资源包不得包含：

- API Key。
- 私有 Base URL，除非用户明确选择。
- 设备标识。
- 无障碍服务状态。
- 本地绝对路径。

### 14.2 HTML / JS 谨慎

如果未来参考叙说做模板：

- 不建议第一版执行 JS。
- 不建议直接加载不可信 HTML。
- 需要沙箱。
- 需要权限说明。
- 需要禁用网络或限制能力。

原生 Android 更适合先做结构化卡片，而不是执行网页脚本。

### 14.3 Room 迁移

新增信箱、资源包索引、主题包等都会影响数据库。

必须遵守：

- Room 版本一次只 +1。
- migration 连续。
- schema 测试保留。
- 不跳版本。

### 14.4 Prompt 安全

角色扮演提示词要保留沉浸感，但默认开关要谨慎。

建议：

- 深度沉浸、反套路、网络热梗等作为高级设置。
- 默认不要引导用户越过平台或法律边界。
- 用户可见文案保持中性。

## 20. 最终建议

Narra 应该学习叙说的「完整度」，但不要学习成叙说的复制品。

最稳的方向是：

1. 先把已有功能入口梳理清楚。
2. 把沉浸主壳做成核心体验。
3. 给 Narra 一个自己的默认视觉。
4. 把导入导出升级成资料库。
5. 新增信箱作为第一批真正的新模块。
6. 最后再考虑社区、音乐、小说、论坛等大模块。

一句话路线：

先把 Narra 做成「好用、完整、有自己气质的原生沉浸角色 App」，再考虑做「大而全生态」。
