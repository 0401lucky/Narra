# Narra 发布与更新约定

## 1. 渠道

- 正式渠道：`release`
- 开发渠道：`dev`
- 基线渠道：`baseline`

对应包名：

- `release` -> `com.narra.app`
- `dev` -> `com.narra.app.dev`
- `baseline` -> `com.narra.app.baseline`

---

## 2. 版本比较

客户端后续检查更新时：

- 只使用 `versionCode` 判断新旧
- `versionName` 只负责展示

当前规则：

```text
versionCode = major * 10000 + minor * 100 + patch
```

---

## 3. APK 文件名

统一输出格式：

```text
Narra-v{versionName}-{versionCode}-{channel}.apk
```

示例：

- `Narra-v1.0.0-10000-release.apk`
- `Narra-v1.0.0-dev-10000-dev.apk`
- `Narra-v1.0.0-baseline-10000-baseline.apk`
- 当前开发分发包示例：`Narra-v1.6.7-dev-10607-dev.apk`

---

## 4. 服务端更新元数据

建议按渠道分别维护：

- `release.json`
- `dev.json`
- `baseline.json`

推荐字段：

```json
{
  "app_id": "com.narra.app",
  "channel": "release",
  "latest_version_name": "1.1.0",
  "latest_version_code": 10100,
  "minimum_supported_version_code": 10000,
  "apk_url": "https://example.com/narra/Narra-v1.1.0-10100-release.apk",
  "apk_sha256": "必填，填写 APK 的 SHA-256",
  "published_at": "2026-03-22T12:00:00+08:00",
  "release_notes": [
    "新增功能 A",
    "修复问题 B"
  ]
}
```

说明：

- 当前客户端统一按“可选更新”处理，不启用强制更新。
- `minimum_supported_version_code` 仅作为兼容保留字段，建议填写 `0` 或不高于当前已发布版本。
- 当 `latest_version_code` 高于本地版本时，`apk_url` 和 `apk_sha256` 都是必填字段；缺少校验值会被客户端拒绝。
- dev 包默认上传到 Cloudflare R2 的 `narra-updates/dev/` 目录，并通过 `https://download.lsa1230.dpdns.org/dev/...` 分发。
- 使用 `wrangler r2 object put ... --remote --content-type application/vnd.android.package-archive` 上传，避免误写本地模拟资源或 MIME 类型不正确。
- 若自定义下载域名曾缓存过 404，可在 `apk_url` 后追加版本查询参数，例如 `?v=10607`。

---

## 5. 发布前验证建议

每次更新 `dev.json` 前建议至少完成：

1. `.\gradlew.bat testDebugUnitTest --console=plain`
2. `.\gradlew.bat assembleDebug --console=plain`
3. `Get-FileHash` 校验 APK SHA-256，并写入 `apk_sha256`
4. 用 `wrangler r2 object get --remote` 拉回远端对象并再次校验 SHA-256
5. 用 `Invoke-WebRequest -Method Head` 确认公开 APK URL 返回 `200`
6. 确认失败重试与渠道隔离逻辑没有被本次改动破坏
