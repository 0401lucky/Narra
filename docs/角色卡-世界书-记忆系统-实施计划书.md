## 0. 你当前工程的真实现状（作为可行性基线）

以下结论来自对当前仓库代码结构的核对（重要：计划不会要求推倒重来）：

### 0.1 入口与装配
- `app/src/main/java/com/example/myapplication/ChatApplication.kt`
  - 在 `onCreate()` 手工装配：
    - `ChatDatabase`（Room，version=7，已具备迁移体系 MIGRATION_1_2 ... MIGRATION_6_7）
    - `AppSettingsStore`（DataStore Preferences + SecureValueStore）
    - `ConversationRepository`（本地会话/消息）
    - `AiRepository`（OpenAI Compatible 网络请求 + SSE 流式）
- `app/src/main/java/com/example/myapplication/ui/AppRoot.kt`
  - 通过 `ChatViewModel.factory()`、`SettingsViewModel.factory()` 注入 `AiRepository`/`ConversationRepository`。

**结论**：你现在没有 DI 框架，但装配集中在 `ChatApplication` 与 `AppRoot`，非常适合“继续用手工装配”的方式新增 WorldBook/Memory 等仓库。

### 0.2 会话与角色（Assistant）现状
- `Assistant` 目前存在于：
  - `model/Assistant.kt`（Data class + 内置 assistants）
  - `AppSettings.assistants`（DataStore JSON 持久化）
  - `Conversation.assistantId`（Room conversations 表字段）
- `ChatViewModel.resolveSystemPrompt()`：当前**只**返回 `assistant.systemPrompt`。
- `ConversationRepository.observeConversationsByAssistant()`：已支持按 assistant 分组会话。

**结论**：
- “角色卡”天然可以扩展 `Assistant`（最小改动路径）。
- “世界书/记忆”更适合独立 Room 表，并通过 assistantId / conversationId 做 scope。

### 0.3 发送链路（关键可行性点）
- `ChatViewModel.submitOutgoingMessage()` 在发请求前会拿到：
  - `systemPrompt`（来自 assistant）
  - `requestMessages`（当前会话已完成消息 + 本轮 user 消息）
- `AiRepository.sendMessageStream(messages, systemPrompt)` 会：
  - `toRequestMessages(messages, systemPrompt)`
  - 把 `effectiveSystemPrompt` 作为一条 `role=system` 注入
  - 自动追加默认格式化提示与玩法提示（`DefaultChatFormattingPrompt` + `DefaultSpecialPlayPrompt`）

**结论**：
- 你只要把“世界书/记忆/摘要”最终组装为一个 `systemPrompt` 字符串（或多个块拼接成一个字符串），就能无侵入接入网络层。
- 因此新增一个 `PromptContextAssembler` 放在 ViewModel 与 Repository 之间，是最可行且改动可控的方案。

## 1. 总体设计原则（确保不变成“专用小手机”）

### 1.1 对外命名：保持“通用聊天 App”语义
建议 UI 文案统一使用：
- 角色卡 → 「助手高级设定」/「角色设定」
- 世界书 → 「知识设定」/「背景设定」
- 记忆 → 「记忆与摘要」

避免将“酒馆/小手机”作为主入口文案，以免产品观感变成小众工具。

### 1.2 架构原则：职责清晰
- `AiRepository`：保持“网络请求 + DTO 解析”纯粹，不混入命中/检索规则。
- `ChatViewModel`：负责编排流程（状态、落库、调用组装器、调用 repository）。
- 新增 `context/`：承接“世界书命中、记忆选择、摘要拼接、最终 prompt 生成”。

### 1.3 存储原则：DataStore vs Room 的边界
- DataStore（`AppSettingsStore`）：适合“用户设置 + 少量结构（助手列表）”。
- Room（`ChatDatabase`）：适合“可增长的条目集合（世界书/记忆/摘要）”。

## 2. 功能定义（与酒馆相似但更通用）

### 2.1 角色卡（Assistant 扩展）
角色卡目标：让不同助手具备稳定人格/写法/世界观入口。

**新增字段建议（第一版）**：
- `scenario: String`：场景/背景（可空）
- `greeting: String`：开场白（可空）
- `exampleDialogues: List<String>`：示例对话（可空，先用字符串列表/多段文本即可）
- `creatorNotes: String`：作者备注（仅编辑页显示，不注入模型）
- `linkedWorldBookIds: List<String>`：关联世界书（可空）
- `memoryEnabled: Boolean`：是否启用记忆注入
- `memoryMaxItems: Int`：每轮最多注入多少条记忆（默认 6）
- `worldBookMaxEntries: Int`：每轮最多注入多少条世界书条目（默认 8）
- `contextTokenBudgetHint: Int`（可选）：软预算提示（先不做精确 token 计算，按字符/条数限制）

**为什么可行**：
- `Assistant` 已通过 Gson JSON 持久化，新增字段向后兼容（旧 JSON 无字段会走 Kotlin 默认值）。
- `AssistantEditScreen` 已存在，可直接扩展 UI。

> 注意：当前 `AppSettingsStore.decodeAssistants()` 没有 normalize，但新增字段只要提供默认值即可，风险很低。若你担心 null，可在 Phase 1.2 增加一次 normalize（见后文）。

### 2.2 世界书（WorldBook）
世界书目标：让模型在提到“人物/地名/组织/规则”时能自动获得相关设定，但不会每轮全塞进上下文。

**世界书条目（Entry）**：
- title：条目标题
- content：条目内容
- keywords：命中关键词（逗号/数组）
- aliases：别名（可选）
- alwaysActive：是否永远注入
- priority：优先级
- enabled：启用
- scope：全局/按 assistant/按 conversation

**命中策略（第一版可行的简化策略）**：
- 输入文本 =（本轮 user 文本 + 最近 N 条 user/assistant 的纯文本拼接）
- 对每条 entry：
  - alwaysActive 直接命中
  - 否则关键词任意命中（contains/Regex 可选）
- 排序：alwaysActive 优先，其次 priority，最后 updatedAt
- 截断：最多注入 `assistant.worldBookMaxEntries`

> 第一版不做 embedding/语义搜索，降低复杂度，易调试。

### 2.3 记忆系统（Memory + Summary）
目标：解决“聊久了忘”和“关系设定断层”。

拆成两件事：

#### A) 结构化记忆（MemoryEntry）
- 用户事实、偏好、关键剧情点
- 支持 pinned（强制注入）
- 支持 scope：global/assistant/conversation

注入策略（第一版）：
- pinned 优先
- 然后按 importance + 最近使用时间
- 限制条数：`assistant.memoryMaxItems`

#### B) 会话摘要（ConversationSummary）
- 用于长对话压缩

触发策略（第一版）：
- 当会话已完成消息数 > 阈值（例如 40 条）
- 对较早消息生成摘要，存入表
- 请求时注入：summary + 最近 K 条消息

**可行性依据**：
- 你现在已经有 `ConversationRepository.listMessages()` / observe 流
- `AiRepository` 已有 `generateTitle()`、`generateChatSuggestions()` 这类“用指定模型做辅助生成”的先例

## 6. Prompt 拼装规范（第一版可行且可控）

### 6.1 System Prompt 模板建议（拼接为一个字符串）
建议在 `PromptContextAssembler` 内按如下顺序组装（每段之间空两行）：

1) 角色卡 persona（原 `assistant.systemPrompt`）
2) 场景（scenario）
3) 世界书命中条目（按条目列表）
4) 记忆（按条目列表）
5) 可选：开场白（greeting）

注入格式建议：

- 世界书：
```
【背景设定（世界书）】
- 条目1标题：条目1内容
- 条目2标题：条目2内容
```

- 记忆：
```
【已知信息（记忆）】
- ...
```

### 6.2 为什么只用一个 system message 也可行
因为 `AiRepository.buildChatSystemPrompt(customPrompt)` 会把你的 `customPrompt` 与默认格式化提示拼在一起，最终仍是一条 system message。

> 未来若你想更“结构化”，也可以改 `AiRepository.toRequestMessages()` 支持多个 system messages，但第一版没必要动网络层。

## 8. 记忆与摘要：第一版的实现策略（可落地）

### 8.1 手动记忆（最先落地，性价比最高）
在 `MessageBubble` 的长按菜单（你现在已经有复制/分享/翻译/重试等能力）里加：
- 「记住这条」：将选中的 message 内容存入 `memory_entries`（scope=conversation 或 assistant）
- 「取消记忆」：删除对应 memory entry（若你用 sourceMessageId 做唯一关联）

> 第一版记忆只做手动，稳定、可控、易用。

### 8.2 自动摘要（次优先，但解决长对话失忆）
在 `ChatViewModel.executeAssistantRoundTrip()` 完成落库后：
- 检查会话 completed 消息数
- 超过阈值且 `coveredMessageCount` 落后，则触发摘要生成

摘要生成方式：
- 直接复用 `AiRepository.sendMessage()` 用一个小 prompt
- 使用 `ProviderFunction.TITLE_SUMMARY` 类似的“功能模型”选择机制（你已存在）

> 这块属于 Phase 3，计划里写清楚，但不强制你 Phase 1 就上。

## 11. 风险点与规避（可行性保证）

### 11.1 DataStore assistants 体积风险
- 角色卡字段变多会让 `assistants_json` 增大。
规避：
- 大文本（大量世界书内容/记忆）不要放 Assistant；放 Room。
- Assistant 里只保存关联 id 与少量文本（scenario/greeting）。

### 11.2 token/上下文膨胀风险
规避：
- worldBookMaxEntries / memoryMaxItems
- content 预截断（例如每条最多 800-1200 字符）
- summary 控制长度（例如 800-1500 字符）

### 11.3 迁移失败风险
规避：
- MIGRATION_7_8 只新增表，不动旧表
- 写好 `IF NOT EXISTS`
- 升级后跑一次 `assembleDebug` + 启动验证

### 11.4 职责污染风险
规避：
- 命中/筛选全部放 `context/`
- `AiRepository` 不改（第一版）

## 12. 实施顺序（你照着做就能落地）

> 这是“最小阻力路径”，按顺序做，每一步都能编译运行。

1) 扩展 `Assistant` 字段（只改 model）
2) 扩展 `AssistantEditScreen` UI（只改 settings UI）
3) 新增 `PromptContextAssembler`（先只拼 systemPrompt + scenario）
4) ChatViewModel 接入 assembler（确保发送仍通）
5) Room：升级到 version=8 + MIGRATION_7_8（新增世界书/记忆/摘要表）
6) 实现 WorldBookRepository + Matcher，接入 assembler
7) UI：新增世界书列表/编辑页，打通 CRUD
8) 手动记忆：MessageBubble 增加入口，MemoryRepository CRUD
9) 摘要（可选）：异步生成并注入
10) 单测补齐（assembler/matcher 优先）

## 14. 本计划的“最关键可行性证明”一句话

你现在的网络层已经支持：`sendMessageStream(messages, systemPrompt)` 并把 `systemPrompt` 注入为 system message。
因此只要我们在 `ChatViewModel` 发请求前增加一个 `PromptContextAssembler` 来拼 `systemPrompt`，再用 Room 提供世界书/记忆数据源，就能稳定落地；而且不会破坏你现有“通用聊天 App”结构。
