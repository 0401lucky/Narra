## 1. 产品定位：它不是酒馆，也不是完整游戏引擎，而是“AI 视觉对话模块”

### 1.1 推荐对外命名
建议不要直接叫“酒馆模式”，而是叫：
- **沉浸扮演**
- **剧情互动**
- **视觉对话**
- **角色剧情**
- **AI 互动小说**

### 1.2 产品边界（很重要）
第一版不要把它定义成：
- 通用游戏编辑器
- 复杂分支剧情引擎
- 多角色站位系统
- Live2D / 动画演出系统
- 重脚本可视化编辑器

第一版应该定义成：

> “一个具有背景图、用户立绘、角色立绘、剧情文本框、旁白区和 AI 连续对话能力的沉浸式互动界面。”

### 1.3 模块关系
建议 App 结构明确分成两条线：

#### A. 普通聊天模块
- 继续保留现在的聊天页
- 用于通用 AI 问答 / 多助手 / 多模型

#### B. 沉浸扮演模块
- 用于视觉小说式对话
- 强调沉浸感、场景感、角色一致性、剧情连续性

这样不会污染主聊天路径，也不会把整个 app 弄成“小手机专用”。

## 3. 模块能力定义（第一版）

### 3.1 用户可配置内容
第一版建议支持用户自定义：

#### 场景层
- 场景标题
- 场景简介
- 背景图（本地 Uri）
- 初始场景说明

#### 用户层
- 用户显示名
- 用户立绘 / 头像（本地 Uri 或 URL）
- 用户定位（左/右，默认左）

#### 角色层
- 关联一个已有 Assistant
- 角色显示名（允许覆盖 Assistant.name）
- 角色立绘 / 头像（默认取 Assistant.avatarUri，可覆写）
- 初始出场台词（可空，默认用 Assistant.greeting）

#### 沉浸参数层
- 是否显示旁白区
- 是否显示角色名条
- 是否自动高亮当前说话方
- 是否启用 RP 模式输出协议
- 是否允许背景切换（先预留，第一版可不做）

### 3.2 页面表现
RP 页面建议结构：

1. **全屏背景图**
2. **左侧人物立绘（通常为用户）**
3. **右侧人物立绘（通常为角色）**
4. **中间/顶部可选场景标题**
5. **底部对话框**
   - 说话者名称
   - 台词文本
6. **可选旁白区**
7. **底部输入栏 / 选项按钮区**

### 3.3 消息类型（第一版）
RP 模块中至少需要区分：
- `narration`：旁白
- `dialogue_user`：用户说话
- `dialogue_character`：角色说话
- `system`：系统提示（如“场景已切换”）

> 第一版不强制支持 thought/inner monologue/多人对话。

## 4. 架构方案：新增 RP 领域层，但复用现有聊天核心

### 4.1 建议总体架构

```text
RP Screen / RP Setup Screen
        ↓
RoleplayViewModel
        ↓
RoleplayRepository
        ↓
ConversationRepository + AiRepository + PromptContextAssembler
        ↓
Room / DataStore / OpenAI Compatible API
```

### 4.2 关键思想
- **ConversationRepository 继续负责底层真实消息存储**
- **RP 模块只是在显示层和数据组织层上“包装”这些消息**
- 一个 RP 场景可以绑定一个 conversationId
- RP 模块不需要自己再造一套 AI 发送引擎

### 4.3 两种实现路线比较

#### 路线 A：完全复用 ChatMessage / Conversation（推荐）
做法：
- RP 模块只新增 `RoleplayScenario` / `RoleplaySession`
- 底层仍使用 `Conversation` / `ChatMessage`
- AI 回复内容经过 RP 协议解析，再用于 UI 展示

优点：
- 改动最小
- 现有记忆/摘要/上下文/导入导出能力直接兼容
- 便于维护

缺点：
- RP 的展示语义与通用消息语义之间需要做一层转换

#### 路线 B：RP 独立消息表
做法：
- 新增 `RoleplayMessageEntity`
- 完全独立一套消息系统

缺点：
- 会造成消息、摘要、记忆、上下文装配的双轨维护
- 不建议第一版采用

**结论**：
第一版务必选 **路线 A**。

## 9. RP 输出协议（这决定沉浸感能不能稳定）

### 9.1 为什么一定要有协议
如果让模型自由输出一整段大文本，前端就无法稳定区分：
- 这是旁白
- 这是角色说话
- 这是情绪
- 这是场景变化

因此第一版至少要有一个**轻量可解析协议**。

### 9.2 第一版协议建议：极简 XML / 标签协议
建议只支持以下两类：
```xml
<narration>夜色慢慢沉下来，窗外的雨声更清晰了。</narration>
<dialogue speaker="character" emotion="calm">你终于来了。</dialogue>
```

用户输入则天然视为：
- `speaker=user`
- `type=dialogue`

### 9.3 第一版不要支持的内容
暂不支持：
- 多角色同时发言
- 复杂站位切换
- 多层嵌套 XML
- 动作脚本 DSL
- Live2D 表情驱动

### 9.4 AI System Prompt 约束建议
在 RP 场景启用时，除了已有 `PromptContextAssembler` 产出的角色卡/世界书/记忆系统提示外，再追加一段 RP 协议提示，例如：

```text
你正在一个沉浸式视觉对话场景中扮演角色。
请使用以下输出规则：
1. 叙述性内容必须使用 <narration>...</narration>
2. 角色对白必须使用 <dialogue speaker="character" emotion="情绪">...</dialogue>
3. 不要解释标签本身
4. 不要输出多余 Markdown
5. 每次回复最多包含 1-3 段 narration/dialogue
```

> 这段最好由 RP 模块自己的 `RoleplayPromptDecorator` 提供，而不是污染通用聊天 prompt。

## 11. 现有功能如何移植到 RP 模块（这是本次计划最关键的一部分）

你已经完成的功能可以这样迁移：

### 11.1 Assistant → RP 角色基础
复用方式：
- `assistantId` 作为 RP 场景绑定角色
- `Assistant.name` 作为默认角色名
- `Assistant.avatarUri` 作为默认角色图
- `Assistant.systemPrompt/scenario/greeting/exampleDialogues` 继续参与 PromptContextAssembler

结论：
> RP 模块不用再做一套“角色卡系统”，直接吃现成 Assistant。

### 11.2 WorldBook → 场景背景设定
复用方式：
- 场景绑定 assistant 时，assistant 已关联 `linkedWorldBookIds`
- `PromptContextAssembler` 在发言前自动命中世界书并注入

如果你后续要更进一步：
- 可以给 `RoleplayScenario` 增加 `linkedWorldBookIds` 覆盖层
- 形成「角色默认世界书 + 场景临时世界书」双层结构

### 11.3 Memory / Summary → 剧情连续性
复用方式：
- 仍由 `PromptContextAssembler` 注入
- RP 模块只负责展示；底层依然是 conversation 的记忆与摘要

结论：
> 这一步是 RP 模块最大的优势：剧情不会像一次性聊天一样断掉。

### 11.4 Context Transfer → 未来导入导出 RP 资源包
你已有 `ContextTransferViewModel` 和导入导出系统。
后续可扩展：
- 导出一个 RP Bundle
  - assistant
  - worldbook
  - memory
  - roleplayScenario

但第一版先不做，避免过度扩展。

## Phase 1：做出可运行的 RP MVP（最高优先级）

先做出一条完整链路：
**场景列表 → 新建场景 → 进入 RP 页面 → 以双立绘 + 背景 + 底部剧情框的形式和 AI 对话**

### 新增内容
- `RoleplayScenario` / `RoleplaySession` model
- Room 表 + `MIGRATION_8_9`
- `RoleplayRepository`
- `RoleplayViewModel`
- `RoleplayScenarioListScreen`
- `RoleplayScenarioEditScreen`
- `RoleplayScreen`
- `AppRoutes` / `AppNavHost` 加 RP 路由

### 暂时不做
- 分支剧情
- 复杂协议
- 场景切换
- 表情切换
- 导入导出
- 自动选项生成

### 验收标准
- 能新建一个场景
- 能选择背景图
- 能设置用户图和角色图
- 能选择一个 Assistant 作为角色
- 能发消息并看到角色在 RP 页面里回复

## 14. 风险与规避策略

### 14.1 风险：RP 模块把现有聊天逻辑搅乱
规避：
- 新建 `RoleplayViewModel`，不要把 RP 逻辑塞进 `ChatViewModel`
- 新建 RP routes，不修改 ChatScreen 主路径

### 14.2 风险：模型输出不稳定
规避：
- Phase 1 先允许纯文本回退
- Phase 2 再上结构化输出协议
- parser 必须有 fallback：解析失败时当普通角色对白显示

### 14.3 风险：图片与背景加载性能
规避：
- 第一版只支持单背景、双立绘静态图
- 不做大动画
- 使用本地 Uri + 现有 Bitmap 加载逻辑即可

### 14.4 风险：场景与 assistant 关系混乱
规避：
- 明确：Assistant 是角色基础人格；RoleplayScenario 是具体剧情场景容器
- 一个 Assistant 可被多个 Scenario 复用

## 15. 最推荐的实际开发顺序（非常重要）

请按这个顺序做，阻力最小：

1. 新增 `RoleplayScenario` / `RoleplaySession` model
2. 新增 RP Room 表与 `MIGRATION_8_9`
3. 新增 `RoleplayRepository`
4. 新增 `RoleplayViewModel`
5. 做 `RoleplayScenarioEditScreen`
6. 做 `RoleplayScenarioListScreen`
7. 做 `RoleplayScreen`（先纯文本展示也行）
8. 打通路由与入口
9. 接入背景/双立绘显示
10. 再做 `RoleplayOutputParser` 与 narration/dialogue 分流

这个顺序的好处是：
- 每一步都能运行
- 每一步都有可见成果
- 不会一上来被协议解析或复杂 UI 卡住

## 16. 最后的产品判断

你现在再做这个模块，时机是对的。

因为如果没有角色卡 / 世界书 / 记忆 / 摘要，这个模块最后很容易只是“换皮聊天”；
但你现在这些都已经有了，所以它有机会变成：

> 一个真正有场景感、有持续设定、有角色一致性、有视觉表达的 AI 剧情互动模块。

而且这条路线比单纯“把普通聊天页越堆越重”更像一个完整产品方向。

---

## 17. 本计划的一句话总结

**建议你把 RP 模块做成：基于已有 Assistant + WorldBook + Memory + Summary + PromptContextAssembler 的“独立视觉对话层”，而不是重新造一套聊天系统。**

这样改动最小、可行性最高、沉浸感最强，也最符合你现在仓库的现状。
