# 沉浸式角色扮演模块 - AI帮写功能计划

- 目标仓库：`D:\code\AndroidStudioprojects\MyApplication`
- 文档路径：`docs/沉浸式角色扮演模块-AI帮写功能计划.md`
- 编写时间：2026-03-21（Asia/Shanghai）
- 目标：在沉浸式剧情模块中增加“AI帮写 / 剧情建议”能力，解决用户不知道下一句该怎么写、剧情推进卡住的问题

---

## 0. 功能一句话定义

在沉浸式剧情页面中新增一个 **AI帮写按钮**：

> 基于当前剧情上下文、角色设定、世界书、摘要与记忆，由 AI 生成 **几条适合用户继续剧情的可发送文本建议**，用户可以一键填入输入框，必要时再手动修改或直接发送。

这个功能的核心不是让 AI 替用户自动演，而是：

> **在用户卡壳时，给用户一个“继续玩下去”的入口。**

---

## 1. 这个功能为什么值得做

沉浸式剧情模块和普通聊天不同，用户经常会出现下面这种状态：

- 想继续剧情，但不知道该说什么
- 不知道现在最合适的推进方式是什么
- 怕一句话把角色关系或剧情方向带偏
- 想维持人设口吻，但自己临时编不出来

所以只给一个输入框是不够的。

需要一个介于：
- **用户自己写**
- **AI 直接接管剧情**

之间的中间层。

这个中间层最适合的形式就是：

### AI帮写 / 剧情建议
- AI 不直接替用户做决定
- AI 只生成“可继续剧情的用户侧文本候选”
- 用户保留最后决定权

这比直接“自动续写”更安全，也更符合 RP 交互体验。

## 2. 功能定位

## 2.1 用户侧名称建议

用户看到的按钮文案建议使用以下之一：
- **AI帮写**（最直接）
- **剧情建议**（更产品化）
- **不知道写什么？**（更口语化）
- **帮我续一句**（更亲和）

推荐：

> 按钮主名称：**AI帮写**  
> 副说明：**生成几条可继续剧情的输入建议**

---

## 2.2 它和普通聊天建议的区别

它不是普通聊天里的“猜你想问什么”，而是：

> **基于当前剧情上下文，为“玩家 / 用户这一侧”生成下一步可输入文本。**

也就是说，AI 生成的是：
- 用户下一句可能会说的话
- 用户下一步可能采取的行动描述
- 用户推动剧情的自然表达

而不是：
- 角色回复
- 剧情旁白
- 系统总结

## 3. 第一版产品设计（推荐最小可用版）

## 3.1 入口位置

建议放在 `RoleplayDialoguePanel` 的输入区附近。

推荐布局：
- 输入框右侧保留发送按钮
- 输入框左侧或上方加一个 **AI帮写** 按钮
- 点击后拉起建议区域

推荐交互：
- 未输入时：AI帮写按钮始终可见
- 已有用户手动输入时：仍可点击 AI 帮写进行参考
- 发送消息后：旧建议自动清空

---

## 3.2 第一版建议输出形式

建议一次生成 **3 条候选文本**，每条都带一个风格标签。

例如：

1. **试探推进**  
   “我看着他，故意放轻了语气：你刚刚那句话，到底是什么意思？”

2. **情绪加压**  
   “我没有退开，只是盯着他问：你是不是早就知道会变成这样？”

3. **信息探索**  
   “我先压下情绪，低声追问：这里到底发生过什么，把你知道的都告诉我。”

这样比只给一条更好，因为：
- 用户有选择空间
- 不会觉得 AI 在强行替自己决定
- 能兼顾不同推进风格

---

## 3.3 点击后的行为

第一版建议采用：

### 默认行为：点击后填入输入框，不自动发送
原因：
- 用户仍可微调措辞
- 更安全
- 更符合“帮写”而不是“代写接管”

### 可选增强行为：长按 / 二级按钮直接发送
第一版可以不做，避免复杂度增加。

---

## 3.4 建议文本的写法原则

AI 生成的建议必须满足：

1. **站在用户 / 玩家视角写**
2. **不能替角色发言**
3. **不能写成旁白总结**
4. **要能自然接在当前剧情后面发送出去**
5. **长度适中，第一版建议 1~3 句**
6. **每条建议风格要有差异**

建议分成三类风格：
- 推进剧情
- 探索信息
- 拉动关系 / 情绪

## 4. 推荐的交互流程

## 4.1 基本流程

1. 用户进入 RP 场景
2. 用户看完当前角色回复，不知道下一句怎么写
3. 点击 **AI帮写**
4. 系统进入建议生成中状态
5. 展示 3 条候选文本
6. 用户点击其中一条
7. 候选文本填入输入框
8. 用户可以：
   - 直接发送
   - 稍微修改再发送
   - 重新点 AI帮写刷新建议

---

## 4.2 建议状态管理

建议区至少需要三种状态：

### A. 空状态
- 尚未请求建议
- 展示按钮或提示语：
  - “不知道写什么？让 AI 帮你续一句”

### B. 加载状态
- 展示 loading
- 文案例如：
  - “正在根据剧情生成输入建议…”

### C. 结果状态
- 展示 3 条建议卡片
- 支持：
  - 点击填入
  - 重新生成
  - 收起

## 5. 技术实现原则

## 5.1 这不是消息发送，而是“建议生成”

这个功能生成的文本：
- **默认不写入 conversation 历史**
- **只有用户真正发送后才进入消息记录**

否则会污染剧情历史。

### 原则
- 建议是“临时 UI 状态”
- 不是正式消息

---

## 5.2 必须复用现有 RP 上下文能力

建议生成不能另起炉灶，应该复用已有：
- `PromptContextAssembler`
- `RoleplayPromptDecorator`
- `RoleplayTranscriptFormatter`
- 摘要
- 世界书
- 记忆
- 当前场景信息

否则生成出来的建议会脱戏。

---

## 5.3 第一版不需要新建独立会话

建议生成应使用：
- 当前 scenario
- 当前 session 对应的 conversation
- 当前 assistant
- 当前上下文拼装结果

也就是说，它只是：

> **基于当前剧情，生成用户侧下一句建议**

而不是创建新的 AI 对话链。

---

## 5.4 请求生命周期与并发控制

`AI帮写` 不是一次性的纯静态按钮，它需要明确的协程与状态管理规则。

第一版建议增加：

- `suggestionJob`，用于跟踪当前建议生成任务
- 若用户在生成中再次点击 `AI帮写` / `换一批`，应先取消旧任务，再启动新任务
- 若用户离开场景、重开剧情、清空剧情、发送新消息，应取消当前建议生成任务
- 若用户在生成过程中开始手动输入，也建议取消当前生成请求，避免浪费 token 和状态错乱

原则：
- 同一时刻仅允许一个 suggestion 请求在飞行中
- UI 状态更新必须以“当前场景 / 当前 session 仍然有效”为前提

---

## 5.5 建议生成的上下文窗口控制

建议生成虽然要参考完整剧情设定，但**不应该无上限塞入全部历史记录**。

推荐策略：

- 系统上下文仍复用：
  - `PromptContextAssembler`
  - 摘要
  - 世界书
  - 记忆
  - 场景与角色设定
- 剧情转写文本只取最近 **5~10 轮消息**
- 若已有摘要，优先保留摘要 + 最近消息窗口，而不是把全部历史再次拼进去

原因：

- 降低 token 消耗
- 减少延迟
- 避免模型被过长历史分散注意力
- 让建议更贴近“当前这一拍”剧情推进

## 6. 推荐实现方案

## 6.1 数据模型建议

新增模型：

### `RoleplaySuggestionUiModel.kt`
建议放在：
- `app/src/main/java/com/example/myapplication/model/RoleplaySuggestionUiModel.kt`

建议结构：

```kotlin
data class RoleplaySuggestionUiModel(
    val id: String,

```

例如：
- `label = "试探推进"`
- `label = "情绪施压"`
- `label = "信息探索"`

---

## 6.2 ViewModel 状态新增

在 `RoleplayUiState` 中新增：

```kotlin
val suggestions: List<RoleplaySuggestionUiModel> = emptyList()
val isGeneratingSuggestions: Boolean = false
val suggestionErrorMessage: String? = null
```

---

## 6.3 ViewModel 新增方法

在 `RoleplayViewModel` 中新增：

```kotlin
fun generateSuggestions()
fun applySuggestion(text: String)
fun clearSuggestions()
```

### 行为说明

#### `generateSuggestions()`
- 校验当前场景、session、assistant 是否存在
- 读取当前 conversation 消息
- 通过 `RoleplayTranscriptFormatter` 转为可读剧情文本
- 调用 suggestion 生成接口
- 解析结果填充到 `suggestions`

#### `applySuggestion(text)`
- 将建议文本写入 `input`
- 可选是否清空建议列表

#### `clearSuggestions()`
- 在用户发送后、退出场景后、重开剧情后清空

---

## 6.4 Repository / AI 层建议

推荐在 `AiRepository` 中新增一个独立方法：

```kotlin
suspend fun generateRoleplaySuggestions(
    conversationExcerpt: String,
    systemPrompt: String,
    baseUrl: String,
    apiKey: String,
    modelId: String,
): List<RoleplaySuggestionResult>
```

如果不想再建新 model，也可以直接返回：

```kotlin
List<RoleplaySuggestionUiModel>
```

### 结构化输出建议

第一版建议采用：

- **优先兼容所有当前 provider 的做法**：通过强约束 prompt 要求返回 JSON
- Kotlin 侧使用本地 JSON 解析
- 解析失败时做安全 fallback，保证不 crash

也就是说，第一版不要把 `Function Calling / Tool Calling / 强制 JSON Mode` 设为硬前置条件。

原因：

- 当前项目走的是通用 `OpenAI-compatible` 接口
- 不同 provider 对 `tool calling`、`json schema`、`response_format` 的支持并不稳定
- 若把这些能力作为第一版硬依赖，会降低兼容性

更稳妥的分层做法是：

### 第一版
- 强约束 JSON prompt
- 本地解析
- fallback

### 后续增强
- 若当前 provider 明确支持 `JSON Mode` 或 `Tool Calling`
- 可升级为更严格的结构化输出，进一步降低解析失败率

## 7. Prompt 设计建议

## 7.1 目标
让模型输出的是：
- **用户下一步可发送文本**
- 不是角色回复
- 不是剧情总结
- 不是分析解释

---

## 7.2 建议的提示词结构

系统提示可类似：

```text
你是沉浸式剧情输入建议生成器。
你的任务不是继续扮演角色，而是基于当前剧情上下文，为“玩家/用户这一侧”生成 3 条可发送的下一步输入建议。

要求：
1. 所有建议都必须站在用户视角书写。
2. 不能替角色发言，不能写角色回复。
3. 每条建议都必须能自然接在当前剧情后面，作为用户下一条输入发送出去。
4. 必须尽量保持玩家最近几轮的说话口吻与人设一致；如果玩家此前偏冷静、强势、试探或克制，建议也要延续该风格。
5. 建议风格要有差异，至少覆盖：推进剧情、探索信息、推动情绪/关系。
6. 建议内容可以同时包含“动作描写”和“对话”；若包含动作描写，优先使用当前项目常见的写法，例如 *动作*。
7. 每条建议控制在 1~3 句，避免过长。
8. 不要输出 Markdown，不要输出解释。
9. 只输出 JSON 数组。

输出格式：
[
  {"label":"试探推进","text":"..."},
  {"label":"信息探索","text":"..."},
  {"label":"情绪拉扯","text":"..."}
]
```

---

## 7.3 输入内容建议

建议生成时输入给模型的内容至少包括：
- 当前场景标题
- 当前场景描述
- 当前角色名
- 当前用户显示名
- 当前摘要
- 当前记忆
- 当前剧情近几轮转写文本
- 当前 RP prompt（可选）

推荐使用：
- `PromptContextAssembler.assemble(..., promptMode = PromptMode.ROLEPLAY)`
- `RoleplayTranscriptFormatter.formatMessages(...)`

然后在建议 prompt 中组合为：

```text
【剧情上下文】
...

【最近剧情】
...

请基于以上内容生成 3 条“用户下一步输入建议”。
```

说明：

- 若当前输入状态仍为简单字符串，第一版可先实现“填入但不自动发送”
- 更细的光标控制可在后续把输入状态升级为更适合控制选区的结构

## 11. 验收标准

实现完成后，至少满足以下标准：

### 功能正确性
- [ ] 点击 AI帮写后能生成 3 条建议
- [ ] 建议文本明显站在用户视角
- [ ] 建议文本不是角色回复
- [ ] 建议文本可以自然接在当前剧情后发送
- [ ] 点击建议后是填入输入框，而不是直接污染历史消息

### 上下文一致性
- [ ] 建议会参考当前剧情历史
- [ ] 建议会参考当前角色与场景设定
- [ ] 有摘要 / 记忆时，建议质量明显比无上下文更稳定

### 体验效果
- [ ] 用户卡壳时有明确可用入口
- [ ] 建议风格不是 3 条完全重复的废话
- [ ] 支持“换一批”
- [ ] 发送后不会残留过期建议

### 边界与异常处理
- [ ] 弱网或请求超时下，UI 能正确从 loading 恢复，并提示“生成失败，请重试”
- [ ] AI 返回非预期格式时，解析异常会被正确捕获，不会导致应用崩溃
- [ ] 在 AI 生成中途，用户若手动发送新消息、退出场景、重开剧情或清空剧情，后台生成任务会被正确取消
- [ ] 多次快速点击 `AI帮写` 或 `换一批` 不会触发并发请求重叠
- [ ] 历史建议不会污染正式消息记录
- [ ] 有摘要时，建议生成不会再把全部历史重复拼装，避免无意义 token 膨胀

若后续要进一步优化等待体验，可升级为：

- 流式 suggestion 生成
- 先展示第一条，再逐步补齐后两条
- 或按卡片逐张浮现，降低等待焦虑

这不是第一版必须项。

---

## 13. 最终一句话总结

**这个功能非常适合沉浸式剧情模块，而且优先级其实不低。**

因为它解决的是 RP 场景里的一个真实高频问题：

> 用户不是不想玩，而是经常不知道下一句该怎么写。

最推荐的第一版做法是：

> **做成“AI帮写 / 剧情建议”按钮，基于当前剧情生成 3 条用户侧输入建议，点击填入输入框，不默认自动发送。**

这样实现成本不高、和现有架构兼容度高、用户价值也非常直接。
