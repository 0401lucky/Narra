# Narra 版本策略

- 正式英文名：`Narra`
- 正式 `applicationId`：`com.narra.app`
- 当前开发分发基线版本：`1.1.8`
- 当前开发分发 `versionCode`：`10108`

---

## 1. 版本命名规则

采用语义化版本：

- `major.minor.patch`

含义：

- `major`
  - 重大版本
  - 代表正式里程碑、架构升级或存在明显行为变化
- `minor`
  - 功能版本
  - 新增一批可感知功能时递增
- `patch`
  - 修复版本
  - 修 bug、稳定性修复、小幅体验修正时递增

---

## 2. Android versionCode 规则

为保证安装包升级链路稳定，`versionCode` 采用固定换算：

```text
versionCode = major * 10000 + minor * 100 + patch
```

示例：

- `1.0.0` -> `10000`
- `1.1.0` -> `10100`
- `1.1.3` -> `10103`
- `2.0.0` -> `20000`

---

## 3. 后续升级建议

- 新功能上线：
  - 增加 `minor`
  - 例如：`1.0.0` -> `1.1.0`
- 纯修复发版：
  - 增加 `patch`
  - 例如：`1.1.0` -> `1.1.1`
- 大版本重构或正式阶段跃迁：
  - 增加 `major`
  - 例如：`1.9.4` -> `2.0.0`

每次准备发版时，需同步更新根目录的 `CHANGELOG.md`：

- 未发布改动先写入 `Unreleased`
- 准备正式发版时，再整理到目标版本号下
- 应用内更新展示的 `release_notes` 建议直接基于 `CHANGELOG.md` 生成

---

## 4. 渠道规则

当前固定 3 个构建渠道：

- `release`
  - 应用名：`Narra`
  - `applicationId`：`com.narra.app`
  - `versionName`：例如 `1.0.0`
- `dev`
  - 应用名：`Narra Dev`
  - `applicationId`：`com.narra.app.dev`
  - `versionName`：例如 `1.0.0-dev`
- `baseline`
  - 应用名：`Narra Baseline`
  - `applicationId`：`com.narra.app.baseline`
  - `versionName`：例如 `1.0.0-baseline`

原则：

- 正式对外安装与应用内更新，只使用 `release`
- 日常开发安装，优先使用 `dev`
- 基准性能或专项验证，使用 `baseline`

---

## 5. APK 文件命名规则

构建产物文件名固定为：

```text
Narra-v{versionName}-{versionCode}-{channel}.apk
```

示例：

- `Narra-v1.0.0-10000-release.apk`
- `Narra-v1.0.0-dev-10000-dev.apk`
- `Narra-v1.0.0-baseline-10000-baseline.apk`

这样做的目的：

- 文件一眼能区分渠道
- 文件一眼能区分版本
- 便于后续在应用内更新、外链分发、测试群发包时避免混包

---

## 6. 应用内更新元数据约定

后续若做“应用内提示下载并安装新包”，建议服务端按渠道分别提供元数据。

推荐 JSON 结构：

```json
{
  "app_id": "com.narra.app",
  "channel": "release",
  "latest_version_name": "1.1.0",
  "latest_version_code": 10100,
  "minimum_supported_version_code": 10000,
  "apk_url": "https://example.com/narra/Narra-v1.1.0-10100-release.apk",
  "apk_sha256": "可选，建议提供",
  "published_at": "2026-03-22T12:00:00+08:00",
  "release_notes": [
    "新增沉浸式剧情 AI 帮写",
    "优化建议卡片交互",
    "修复若干稳定性问题"
  ]
}
```

当前客户端默认不启用强制更新：

- 更新弹窗始终允许用户关闭
- `minimum_supported_version_code` 仅保留兼容字段
- 建议配置为 `0` 或不高于当前线上版本

客户端比较规则建议固定为：

- **只比较 `versionCode`**
- `versionName` 仅用于展示

这样能避免：

- `1.0.10` 与 `1.0.9` 的字符串比较错误
- 渠道后缀导致的展示值混淆

---

## 7. 正式包注意事项

当前工程已经固定正式显示名和版本规则。

正式包标识也已固定为：

- `applicationId = com.narra.app`

但若未来要长期做：

- 应用内更新
- 正式签名
- 对外分发

这个值后续不应再随意更改。

原因：

- `applicationId` 一旦用于正式发包，后续继续修改会破坏覆盖安装与应用内更新链路
- 除非明确切换成全新应用，否则应长期保持不变
