# 沉浸式角色扮演模块优化计划书

- 目标仓库：`D:\code\AndroidStudioprojects\MyApplication`
- 文档路径：`docs/沉浸式角色扮演模块-优化计划书.md`
- 编写时间：2026-03-21（Asia/Shanghai）
- 适用对象：用于交给 AI / 开发助手直接按计划进行代码优化
- 目标：解决当前 RP 模块“能用但不够像视觉小说、上下文连续性感知弱、记忆效果不明显、整体体验别扭”的问题

---

## 0. 先说结论

当前沉浸式角色扮演模块并不是“没做出来”，而是已经完成了：
- 独立场景模型
- 独立场景列表 / 编辑 / 游玩页面
- 独立 `RoleplayViewModel`
- 独立 `RoleplayRepository`
- RP 协议提示词与解析器
- 双立绘 + 背景 + 对话面板的表现层

但当前体验仍然偏向：

> **聊天系统外面套了一层 RP 外壳**

而不是：

> **真正稳定、连续、可沉浸的剧情对话系统**

导致用户体感会出现：
- “看起来有 RP 界面，但回复风格不够稳”
- “重新进入场景能续，但我感觉系统没有明确告诉我在续什么”
- “记忆像没生效”
- “整体像双立绘聊天页，不像视觉小说”

本次优化目标不是推翻重写，而是针对以下四个核心层做定向重构：

1. **Prompt 层**：让 RP 提示词不再被普通聊天提示污染
2. **会话层**：让场景 / 会话 / 继续剧情 / 重开剧情语义明确
3. **上下文层**：让摘要、记忆、世界书真正为 RP 服务
4. **表现层**：让交互更像视觉小说而不是聊天页换皮

---

## 1. 当前实现的真实行为说明（必须先统一认知）

## 1.1 当前场景与 session 的关系

当前代码实际行为：
- 一个 `RoleplayScenario` 对应一个 `RoleplaySession`
- 一个 `RoleplaySession` 对应一个底层 `Conversation`
- `roleplay_sessions.scenarioId` 有唯一索引，意味着**一个场景当前只有一条活动剧情会话**
- 再次进入同一个场景时，会优先复用原来的 session / conversation

### 结论
- **重新进入同一个场景，不会丢失上下文**
- 因为底层 conversation 会被复用

但要注意：
- **RP 场景会话** 和 **普通聊天会话** 是两条不同链路
- 用户退出 RP 场景后去普通聊天页继续聊，不会自动接着 RP 场景那条 conversation
- 所以从用户体验上，会感觉“好像上下文断开了”

### 产品语义问题
当前技术上“能续”，但产品上没有明确告诉用户：
- 这是继续旧剧情
- 这是重开剧情
- 这是清空剧情
- 改绑角色后要不要重建剧情

这会直接造成“哪哪不得劲”的感受。

---

## 1.2 当前记忆系统为什么会让人觉得没生效

当前 RP 自动记忆提取逻辑大致是：
- 仅当绑定的 assistant 开启 `memoryEnabled` 时生效
- 仅当当前 provider 可用且 memory model 可解析时生效
- 从最近 6 条消息中抽取记忆
- 记忆全部写入 `MemoryScopeType.ASSISTANT`
- 不写入 `MemoryScopeType.CONVERSATION`
- 不存在“场景级记忆”隔离

### 这会导致两个体验问题

#### 问题 A：同一个角色的多个场景会串记忆
例如：
- 角色 A 在场景 1 里刚经历过某段剧情
- 场景 2 也绑定了角色 A
- 因为记忆是 assistant 级别注入，所以场景 2 也可能拿到场景 1 的剧情残留

#### 问题 B：用户期待“这个场景记住了剧情”，但系统只在做“角色级长期记忆”
这意味着：
- 用户想要的是“这段剧情状态不要丢”
- 当前系统实现的是“角色长时记住一些信息”

这两者不是一回事。

---

## 1.3 当前摘要为什么不明显

当前 RP 摘要生成逻辑偏保守：
- 需要消息数超过一定阈值才触发
- 而且摘要只覆盖较老的历史消息
- 对短测试对话、早期剧情推进帮助不明显

再加上用户界面没有明显显示：
- 当前是否已有摘要
- 摘要覆盖了多少条
- 当前是否注入了记忆

所以用户很容易得出结论：

> “好像根本没生效”

实际上是：
- 有些能力在后台确实存在
- 但**触发门槛高、效果反馈弱、作用域设计不贴合 RP**

---

## 1.4 当前 RP 输出协议存在的实际冲突

当前 RP 模块虽然会在系统提示词中追加：
- `<narration>`
- `<dialogue speaker="character" emotion="...">`

但底层 `AiRepository` 仍会对所有聊天统一追加：
- 默认 Markdown 可读性提示
- 默认特殊玩法 transfer 标签提示

这会导致模型同时收到互相打架的要求：
- 一边要求“输出 RP 标签协议”
- 一边要求“用自然 Markdown 格式回答”
- 一边又提示“支持 transfer 标签玩法”

### 结果
- RP 协议稳定性下降
- 回复风格不纯
- 模型更容易输出“像聊天而不是像剧情”的结果

这属于当前体验异常的**第一优先级根因**。

## 2. 当前主要问题清单（按优先级排序）

## P0 级问题（必须先解决）

1. **RP prompt 被普通聊天 prompt 污染**
2. **场景会话语义不清，用户不知道自己是在继续剧情还是重开剧情**
3. **摘要 / 记忆提取使用了协议原文，导致抽取质量下降**
4. **记忆全部进 assistant 作用域，缺乏剧情状态级记忆**

## P1 级问题（会明显影响稳定性）

5. **改绑角色后仍复用旧 session，容易人格 / 历史错位**
6. **开场旁白永久悬挂在消息顶部，不像真正剧情开场**
7. **解析器允许 `speaker="user"`，模型可能替用户发言**
8. **摘要与记忆触发阈值更适合普通聊天，不适合 RP**

## P2 级问题（会影响“像不像视觉小说”）

9. **UI 仍然是气泡聊天思路，不是 VN / 橙光式思路**
10. **当前高亮逻辑过于简单，旁白时高亮不自然**
11. **缺少上下文状态的可视化反馈**
12. **缺少“剧情继续 / 重置 / 重开”的明确入口**

---

## 3. 优化目标

本轮优化完成后，应达到以下目标：

### 3.1 连续性目标
- 进入同一场景时，用户明确知道自己在“继续剧情”
- 用户可以主动选择“重开剧情”或“清空剧情”
- 改绑角色后，系统能正确提示是否重建剧情会话

### 3.2 上下文目标
- RP 回复只遵守 RP 输出协议，不再被通用聊天提示干扰
- 摘要与记忆提取基于“可读剧情文本”而不是 XML 标签原文
- 剧情状态与角色长期记忆分层管理

### 3.3 体验目标
- 主界面更像剧情对话系统，而不是聊天气泡列表
- 用户能直接感知：摘要命中了多少、记忆注入了多少、当前剧情是否在续
- 角色回复风格更稳定、更沉浸

---

## 4. 优化总原则

1. **不推翻现有 RP 架构**
   - 保留 `RoleplayScenario`
   - 保留 `RoleplaySession`
   - 保留独立 `RoleplayViewModel`
   - 保留底层复用 `ConversationRepository`

2. **优先修正 prompt / context / session 语义**
   - 这些对体感影响最大

3. **尽量复用现有 Memory / Summary / WorldBook 基础设施**
   - 不重复造轮子

4. **第一轮优化重点提升稳定性和可感知性**
   - 先解决“为什么怪”
   - 再追求更强的视觉小说演出

## 5. Phase A：Prompt 与上下文管线重构（最高优先级）

## A.1 为 AI 请求增加“RP 模式”

### 要修改的文件
- `app/src/main/java/com/example/myapplication/data/repository/AiRepository.kt`
- `app/src/main/java/com/example/myapplication/viewmodel/RoleplayViewModel.kt`
- `app/src/main/java/com/example/myapplication/viewmodel/ChatViewModel.kt`

### 方案
为 `sendMessageStream(...)` 增加模式参数，例如：

```kotlin
enum class PromptMode {
    CHAT,
    ROLEPLAY,
}
```

然后在 `AiRepository` 中分开处理：

#### CHAT 模式
继续使用当前：
- `DefaultChatFormattingPrompt`
- `DefaultSpecialPlayPrompt`

#### ROLEPLAY 模式
只使用：
- RP 模块传入的 `RoleplayPromptDecorator.decorate(...)` 结果
- 不再追加默认 Markdown 规则
- 不再追加 transfer 特殊玩法提示

### 目标
彻底消除 RP 协议提示与普通聊天提示的冲突。

### 验收标准
- RP 回复稳定输出 narration / dialogue
- 明显减少 Markdown 标题、列表、聊天口吻漂移
- RP 页面不再出现 transfer 标签玩法内容

---

## A.2 新增“RP 可读文本归一化器”

### 要新增的文件
- `app/src/main/java/com/example/myapplication/roleplay/RoleplayTranscriptFormatter.kt`

### 作用
把 RP 原始内容：

```xml
<narration>夜色渐深。</narration>
<dialogue speaker="character" emotion="calm">你终于来了。</dialogue>
```

转换成适合：
- 摘要生成
- 记忆抽取
- 标题生成
- 调试展示

的可读文本，例如：

```text
旁白：夜色渐深。
角色：你终于来了。
```

### 要接入的位置
- `RoleplayViewModel.buildSummaryInputText(...)`
- `RoleplayViewModel.buildMemoryExtractionInput(...)`
- 可选：调试面板增加“结构化文本预览”

### 目标
避免摘要 / 记忆被 `<narration>`、`<dialogue>` 标签污染。

### 验收标准
- 摘要更像剧情总结而不是标签文本压缩
- 记忆提取更稳定、更像事实和状态归纳

## A.3 给 RP 上下文增加可视化状态反馈

### 要修改的文件
- `app/src/main/java/com/example/myapplication/context/PromptContextAssembler.kt`
- `app/src/main/java/com/example/myapplication/viewmodel/RoleplayViewModel.kt`
- `app/src/main/java/com/example/myapplication/ui/screen/roleplay/RoleplayScreen.kt`

### 方案
在 RP 页面增加一个轻量状态区，显示：
- 当前场景名
- 当前是否存在摘要
- 摘要覆盖消息数
- 世界书命中数
- 记忆注入数
- 当前是否为“继续旧剧情”

### 目标
让用户不再只能靠猜测“记忆是不是没生效”。

### 验收标准
用户能直观看到：
- 当前上下文是否注入成功
- 当前剧情是不是延续了历史

## 6. Phase B：会话生命周期优化

## B.1 明确“继续剧情 / 重开剧情 / 清空剧情”三种语义

### 要修改的文件
- `app/src/main/java/com/example/myapplication/data/repository/roleplay/RoleplayRepository.kt`
- `app/src/main/java/com/example/myapplication/viewmodel/RoleplayViewModel.kt`
- `app/src/main/java/com/example/myapplication/ui/screen/roleplay/RoleplayScenarioListScreen.kt`
- `app/src/main/java/com/example/myapplication/ui/screen/roleplay/RoleplayScreen.kt`

### 当前问题
当前用户只能隐式复用 session，缺乏明确操作语义。

### 方案
增加 3 个明确动作：

#### 继续剧情
- 默认行为
- 进入场景时复用已有 session / conversation

#### 重开剧情
- 删除或替换当前 session 对应的 conversation
- 创建新的 conversation
- 保留场景配置，不保留原剧情历史

#### 清空剧情
- 保留 session 与 conversation 关系
- 清空消息列表
- 可保留部分场景状态 / 配置

### 验收标准
用户在界面上明确知道：
- 我现在是在续旧剧情
- 我现在是在重新开始
- 清空和重开不是一回事

---

## B.2 改绑 assistant 时检测会话失配并提示重建

### 当前问题
场景修改绑定角色后，如果继续复用旧 session，会出现：
- conversation 历史是旧角色
- 当前 prompt 已切到新角色

这会导致人格和剧情错位。

### 方案
在保存场景或进入场景时检查：
- 当前 scenario.assistantId
- 当前 session 对应 conversation.assistantId

如果不一致：
- 弹出提示：
  - 继续使用旧剧情
  - 重建剧情会话（推荐）

### 验收标准
改绑角色后，历史与当前角色人格不会悖离。

---

## B.3 给 session 增加元信息（可选但推荐）

### 方案
如果要增强可解释性，可在 `RoleplaySession` / `RoleplaySessionEntity` 增加字段：
- `assistantIdSnapshot`
- `lastOpenedAt`
- `lastSummaryAt`
- `lastMemoryExtractionAt`

### 数据库影响
如果新增字段，需要：

### 价值
后续更容易判断：
- 当前 session 是否过期
- 是否需要重建
- 记忆 / 摘要最近有没有跑过

## 7. Phase C：RP 记忆机制重构（解决“记忆没生效”的核心）

## C.1 将记忆拆成“角色长期记忆”与“剧情状态记忆”两层

### 当前问题
当前 RP 提取出的记忆全部写入：
- `MemoryScopeType.ASSISTANT`

这会造成：
- 不同场景串剧情
- 场景状态无法隔离

### 方案
利用现有内存模型，不必新增表：

#### 角色长期记忆（Assistant Scope）
适合保存：
- 用户称呼偏好
- 长期关系定位
- 长期个性设定
- 稳定事实

#### 剧情状态记忆（Conversation Scope）
适合保存：
- 当前地点
- 当前任务状态
- 已发生关键事件
- 当前关系推进阶段
- 当前剧情线索

### 接入方式
在 `RoleplayViewModel.launchAutomaticMemoryExtraction(...)` 中：
- 将提取结果分为 persistent / sceneState 两类
- persistent 写入 `ASSISTANT`
- sceneState 写入 `CONVERSATION`

### 验收标准
- 同一角色多个场景不再明显串剧情
- 同一场景重新进入后剧情状态更稳定

---

## C.2 优化自动记忆提取提示词与存储策略

### 当前问题
当前提取逻辑存在几个问题：
- 仅用最近 6 条消息，窗口过小
- 未区分“长期事实”和“场景状态”
- 未设置总量上限，只限制“本轮新增数”
- `sourceMessageId` 未充分利用

### 优化方案

#### 提取输入优化
- 先对 RP 协议内容做归一化
- 输入给记忆模型的不是 XML，而是可读剧情文本

#### 提取输出优化
要求模型分成两类：
1. `persistent_memories`
2. `scene_state_memories`

#### 存储优化
- 角色长期记忆 -> `ASSISTANT`
- 剧情状态记忆 -> `CONVERSATION`
- 为每条记忆写入 `sourceMessageId`
- 增加去重规则：
  - 同 scope 下内容归一化后去重
  - 同 `sourceMessageId + normalizedContent` 去重

#### 上限与淘汰策略
对每个 scope 增加总量控制：
- Assistant 记忆总数上限：例如 20~40
- Conversation 记忆总数上限：例如 12~20

淘汰顺序建议：
1. 保留 pinned
2. 保留 importance 高的
3. 保留最近使用的
4. 淘汰更旧的、低重要性的

### 验收标准
- 记忆不再越积越脏
- RP 中能明显感知到持续记住剧情状态

---

## C.3 调整记忆注入策略

### 当前策略
`MemorySelector` 会在 assistant 开启 memory 时，从所有记忆中按：
- pinned
- importance
- lastUsedAt
- updatedAt

进行排序。

### 优化目标
在 RP 场景下，建议注入优先级改为：
1. 当前 conversation 的剧情状态记忆
2. 当前 assistant 的长期记忆
3. 全局记忆

### 方案
可以保持通用 `MemorySelector` 不变，但新增 RP 专用 selector，或在 selector 中支持模式参数。

### 验收标准
在 RP 中优先注入“当前剧情状态”，而不是离当前场景无关的长期信息。

---

## 8. Phase D：摘要机制优化

## D.1 让 RP 摘要更早触发

### 当前问题
现有阈值更偏向普通聊天，对 RP 来说过晚。

### 建议调整
给 RP 单独一套阈值：
- `SUMMARY_TRIGGER_MESSAGE_COUNT`：降低
- `SUMMARY_MIN_COVERED_MESSAGE_COUNT`：降低
- `SUMMARY_RECENT_MESSAGE_WINDOW`：缩小一点

例如可改为：
- 12
- 4
- 8

### 目标
让剧情在中短回合内也能尽早形成摘要。

### 验收标准
用户在十几轮剧情后，就能开始感知到摘要对连续性的帮助。

---

## D.2 摘要输入必须基于 RP 可读文本

### 方案
摘要输入不再直接使用 assistant 原始 XML 文本，而是使用：
- `RoleplayTranscriptFormatter` 输出的可读剧情文本

### 价值
提升摘要质量，减少摘要内容“像代码 / 像标签文本”的问题。

---

## 9. Phase E：协议与解析器优化

## E.1 禁止模型替用户发言

### 当前问题
`RoleplayOutputParser` 允许：
- `speaker="user"`

这意味着模型一旦跑偏，会替玩家说话。

### 方案
在 assistant 输出解析中：
- 只接受 `character`
- 接受 `narration`
- 对 `speaker="user"`：
  - 直接降级为 `character`
  - 或丢弃并记录调试告警

### 验收标准
模型不再擅自控制用户行为与台词。

---

## E.2 将 openingNarration 改为“首次入场注入”，而不是每次都挂在顶部

### 当前问题
现在 `openingNarration` 在 UI 映射阶段永久存在，导致像“固定装饰文字”，不像剧情真正开场。

### 方案
当场景第一次启动且 conversation 为空时：
- 将 openingNarration 写入一条初始化剧情消息
- 后续不再在 `mapMessages()` 中重复强插

### 验收标准
开场旁白只在故事开始时出现一次，更像真正剧情序章。

---

## E.3 优化流式输出时的显示稳定性

### 问题
流式输出过程中，协议标签可能部分到达，导致中间帧显示不够自然。

### 方案
在流式阶段：
- 对未闭合标签做更稳妥的隐藏 / 清洗
- 或仅显示“当前角色正在组织对白”，直到内容足够成段

### 验收标准
流式输出不再出现半截标签或不自然跳动。

---

## 10. Phase F：UI 与沉浸感优化

## F.1 从聊天气泡流升级为“视觉小说流”

### 当前问题
现有 `RoleplayDialoguePanel` 依然是典型聊天气泡式布局。

### 结果
虽然背景、立绘都有了，但整体仍更像：
- IM 聊天页 + 立绘

而不像：
- 视觉小说 / 互动小说

### 方案
分成两种模式：

#### 默认模式：剧情模式
- 主要展示最新一条 / 最新一段剧情
- 保持底部对白框
- 历史内容不作为主视觉焦点

#### 阅读模式：历史回看
- 保留现有长列表式阅读模式
- 用于回顾剧情历史

### 验收标准
默认进入时更像视觉小说，而不是消息列表。

## F.2 优化立绘高亮逻辑

### 当前问题
当前高亮逻辑主要依赖最后一条消息的 speaker。
如果最后一条是 narration，高亮会不自然。

### 方案
高亮规则改为：
1. 优先找最近一条对白类消息
2. narration 不改变当前高亮说话方
3. 发送中默认高亮角色侧

### 验收标准
立绘高亮更符合剧情演出直觉。

## 13. 如果需要升级数据库，迁移策略如下

如果只是：
- prompt 模式
- 记忆逻辑
- 摘要逻辑
- UI 逻辑

则**不一定需要改数据库结构**。

如果新增以下字段：
- `assistantIdSnapshot`
- `lastOpenedAt`
- `lastSummaryAt`
- `lastMemoryExtractionAt`

则需要：
- `ChatDatabase.version`：`10 -> 11`
- 新增 `MIGRATION_10_11`

### 原则
除非确实需要更强的 session 元信息，否则优先不动 schema，先把体验问题修掉。

## 15. 验收清单（开发完成后逐项确认）

### 会话与连续性
- [ ] 再次进入同一场景时，明确显示“继续剧情”
- [ ] 用户可主动选择“重开剧情”
- [ ] 改绑角色后，进入场景时会提示是否重建剧情会话

### Prompt 与协议
- [ ] RP 回复不再混入普通聊天 Markdown 风格要求
- [ ] RP 回复不再混入 transfer 特殊玩法提示
- [ ] narration / dialogue 输出稳定性提升

### 记忆与摘要
- [ ] RP 摘要能更早生成
- [ ] 摘要内容是剧情文本，不是协议标签文本
- [ ] 记忆分成长期记忆与剧情状态记忆
- [ ] 同一角色不同场景不再明显串剧情
- [ ] RP 页面可看到记忆 / 摘要命中状态

### UI 与体感
- [ ] 默认模式更像视觉小说，不是聊天列表
- [ ] 阅读模式仍可完整查看历史
- [ ] 旁白不会导致立绘高亮错乱
- [ ] openingNarration 只在真正开场时出现一次

## 16. 最终一句话总结

**当前 RP 模块最大的问题不是“缺功能”，而是：**
- Prompt 不够纯
- 会话语义不够清
- 记忆作用域不够对
- 上下文效果不够可感知
- UI 形式上像 RP，交互本质上仍像聊天

**最优解不是推翻重做，而是做一次“Prompt / Session / Memory / UI”的定向重构。**

完成本计划后，RP 模块会从“聊天页换皮”提升为“真正可持续推进剧情的沉浸式角色对话系统”。
